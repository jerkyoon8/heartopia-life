# 🛡️ 하토피아 위키 - 면접 완벽 방어 매뉴얼 (예상 Q&A)

불안해하실 필요 없습니다! 프로젝트의 A부터 Z까지, 설정 파일(`application.properties`)부터 프론트엔드 JavaScript까지 샅샅이 뒤져 **면접관이 물어볼 수밖에 없는 모든 기술적 포인트와 모범 답변**을 정리했습니다. 이 문서만 완벽히 숙지하면 면접에서 절대 당황하지 않습니다.

---

## 1. 서버 성능 및 최적화 (가장 칭찬받을 부분 🌟🌟🌟)

AI가 `application.properties` 파일에 **실무(Production) 수준의 무시무시한 성능 최적화**를 이미 다 해두었습니다. 면접관은 반드시 이 부분을 칭찬하거나 물어볼 것입니다.

### Q. "application.properties 를 보니 최적화를 많이 하셨네요. Gzip 압축은 왜 켜두셨나요?"
* **사용 기술**: `server.compression.enabled=true`
* **모범 답변**: "하토피아 위키는 사진 데이터나 HTML 문서 텍스트가 매우 큽니다. Gzip 압축을 활성화하여 클라이언트와 서버 간 전송되는 **데이터의 크기를 50~70% 절감**시켜, 네트워크 통신 비용과 로딩 시간을 크게 단축했습니다."

### Q. "정적 리소스에 Content Hash 전략을 사용하셨네요. 이게 뭔가요?"
* **사용 기술**: `spring.web.resources.chain.strategy.content.enabled=true`
* **모범 답변**: "이미지나 CSS/JS 파일에 브라우저 캐싱(1년)을 먹여 재방문 속도를 높였습니다. 하지만 파일이 수정되었을 때 사용자 사이트에서 안 바뀌는 문제가 생길 수 있어, 파일 내용(Content) 기반으로 해시(Hash) 값을 URL에 자동으로 붙이는 전략을 써서 **캐시 무효화(Cache Invalidation) 문제**를 해결했습니다."

### Q. "HikariCP(DB 커넥션 풀) 사이즈를 20으로 튜닝하셨는데 이유가 있나요?"
* **사용 기술**: `spring.datasource.hikari.maximum-pool-size=20`
* **모범 답변**: "대규모 트래픽이 아니더라도 검색 통신이 자주 발생할 것이라 판단했습니다. 기본 커넥션(10)보다 약간 늘린 20개로 설정해 병목을 방지했고, 동시에 `idle-timeout`과 `connection-timeout`을 명시하여 안 쓰는 커넥션을 빠르게 반환시켜 DB 메모리 누수를 막고자 했습니다."

## 2. 보안 인프라 (SecurityConfig.java)

관리자 페이지와 일반 위키 페이지를 보호하기 위해 Spring Security가 단단하게 설정되어 있습니다.

### Q. "Spring Security는 어떻게 구성하셨나요? 비밀번호는 어떻게 관리되죠?"
* **모범 답변**: "`SecurityFilterChain`을 통해 지도 조회(GET)와 일반 웹페이지는 모두 열어두고(permitAll), 지도의 핀을 생성/수정/삭제(POST, PUT, DELETE)하거나 관리자 페이지(/wiki/admin/**)로 가는 요청은 **ADMIN(관리자) 권한**이 있어야만 접근 가능하도록 막았습니다. 비밀번호는 당연히 평문으로 저장하지 않고 **BCryptPasswordEncoder**를 사용해 단방향 해시 암호화를 거쳤습니다."

### Q. "관리자 계정 관리는 어떻게 했나요?"
* **모범 답변**: "현재 DB에 사용자 테이블을 별도로 파는 대신 `application.properties` (또는 secret 파일)의 전역 변수에서 어드민 계정/비번을 읽어와 `InMemoryUserDetailsManager`에 주입하는 방식을 택했습니다. 단일 관리자만 존재하는 인디 위키 서비스 특성상 가장 가볍고 안전한 관리 방식이라 판단했습니다."

## 3. 프론트엔드와 백엔드 통신 (JavaScript & REST)

### Q. "지도 핀을 추가(POST)할 때, 403 Forbidden(보안 에러)이 나지 않나요? CSRF 방어는 어떻게 했죠?"
* **사용 위치**: `map-api.js` 내부
* **모범 답변**: "네, 스프링 시큐리티에서 CSRF 방어가 기본 활성화되어 있기 때문에 자바스크립트로 POST 요청 시 에러가 납니다. 이를 해결하기 위해 HTML `<meta>` 태그에 스프링이 발급한 CSRF 토큰을 숨겨두고, `map-api.js`에서 이 토큰을 읽어와 Fetch API 헤더에 넣어서 통신하도록 구현했습니다."

## 4. 백엔드 에러 컨트롤 (GlobalExceptionHandler)

### Q. "서비스 운영 중에 에러가 나면 하얀 에러 페이지가 뜨지 않게 어떻게 방어하셨나요?"
* **사용 위치**: `advice/GlobalExceptionHandler.java`
* **모범 답변**: "`@ControllerAdvice` 애노테이션을 써서 모든 컨트롤러에서 발생하는 에러를 중앙 통제했습니다. 특히, 클라이언트가 던진 헤더(Accept)를 분석해 API 통신 중이면 JSON 포맷으로 깨지지 않게 예외 메시지를 주고, 화면 이동 중이면 예쁜 디자인이 들어간 커스텀 `error.html` 페이지로 이동시키도록 라우팅했습니다."

### Q. "NoResourceFoundException을 굳이 잡아서 아무 작업도 안 하셨던데(무시), 이유가 뭔가요?"
* **사용 위치**: 위 파일 내부의 이미지 NotFound 방어
* **모범 답변**: "도감 특성상 이미지가 등록 안 된 엑스박스 에러(404)가 자주 발생할 수 있습니다. 이미지가 없을 때는 프론트엔드의 `onerror` 로직이 기본 아이콘으로 덮어씌우게 처리했습니다. 굳이 서버에 매번 WARN 로그를 남기면 디스크 낭비와 모니터링 노이즈가 발생하기 때문에 스프링 단에서 404를 무시(`ignoring`)하도록 설계했습니다."

## 5. 데이터 가공 및 MyBatis 쿼리 최적화

### Q. "여러 테이블의 데이터를 조인(JOIN)해야 할 땐 어떻게 하셨나요?"
* **사용 위치**: `MapController.java -> enrichPinDetails()`
* **모범 답변**: "무조건 DB에서 JOIN 쿼리를 쓰면 쿼리가 복잡해지고 인덱스 타기 곤란해집니다. 따라서 지도 핀 정보, 도감 정보 등은 Service에서 각각 단건으로 빠르게 가져온 다음, **Java 8 Stream API**의 `.filter()`와 `.ifPresent()`를 사용해 메모리 애플리케이션 레벨에서 데이터를 매핑(조립)하는 선택을 했습니다."

### Q. "MyBatis 매퍼 사용 시 신경 쓴 점이 있다면요?"
* **사용 위치**: `CollectionMapper.xml`
* **모범 답변**: "중복 코드를 혐오합니다. 모든 SELECT 문에 긴 컬럼명들을 하드코딩하지 않고 `<sql id="fishColumns">` 형태로 조각내어 관리하고, `<include>`를 통해 붙여 넣음으로써 향후 컬럼이 바뀌더라도 한 곳만 수정하면 되도록 유지보수성을 극대화했습니다."
