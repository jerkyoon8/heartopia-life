# Heartopia Wiki 플랫폼 기술 스택 및 아키텍처 리뷰

현재 AI와 협업하여 상당 부분 구현된 프로젝트를 전체적으로 스캔하고 분석한 결과입니다. 현재까지 구현된 기술적 기반과, 앞으로 장기적인 유지보수성 및 안정성을 위해 **필히 개선해야 할 작은 부분들**을 모두 찾아 정리했습니다.

---

## 1. 현재 적용된 핵심 기술 스택
- **Backend Framework**: Java 17, Spring Boot 3.4.2
- **Data Access**: MyBatis (3.0.3), MySQL Connector
- **Frontend Template**: Thymeleaf (서버 사이드 렌더링) + 부분 바닐라 JS/Vue
- **Security**: Spring Security (Thymeleaf 연동 포함)
- **Utilities**: Lombok, Spring Boot Validation

**💡 아키텍처 뼈대 요약:**
- 전형적인 **MVC 아키텍처** (Controller -> Service -> Mapper -> DB) 구조를 잘 따르고 있습니다.
- 복잡한 JPA 대신 **MyBatis를 사용한 SQL 중심적 설계**를 채택하여 쿼리 최적화 직관성이 높습니다.

---

## 2. 구조적으로 개선(리팩토링)이 필요한 발견점들

AI가 요구사항에 맞춰 기능 구현을 우선시하다 보니 설계적으로 놓치거나 빚(Technical Debt)이 쌓인 부분들이 존재합니다.

### 🚨 백엔드(Java/Spring) 측면의 문제점

**1. 컨트롤러의 비대화 (Fat Controller) 및 DTO 분리 부재**
- **현상**: `WikiController` 소스코드를 보면 길이가 600줄이 넘습니다. 특히 컨트롤러 상단에 `Record Fish(...)`, `Bug(...)` 등 화면에 데이터를 던져주기 위한 설계도(DTO)를 컨트롤러 내부에 욱여넣어 두었습니다.
- **개선안**: `com.heartopia.wiki.dto` 패키지를 새로 생성하여, 데이터 전송 객체(DTO)들을 그곳으로 100% 분리해야 합니다. 이렇게 해야 `MapController` 등 다른 곳에서도 재사용할 수 있습니다.

**2. 서비스 레이어의 데이터 변환 로직 누락**
- **현상**: 데이터를 DB에서 꺼내와서 화면용으로 예쁘게 가공하는 작업(예: 가격이 NULL이면 0으로 치환, 날씨가 없으면 '상시'로 치환)을 전부 `Controller`가 하고 있습니다.
- **개선안**: 데이터 가공과 비즈니스 로직은 `CollectionService` 내부로 옮겨야 컨트롤러 코드가 깔끔해지고 유지보수가 쉽습니다.

**3. 캐시 초기화(Cache Eviction) 로직의 부재**
- **현상**: `CollectionService`에 `@Cacheable`이 매우 훌륭하게 적용되어 있어 성능이 빠릅니다. 하지만, **웹상에서 데이터를 추가/수정/삭제했을 때 이 캐시를 날려버리는(`@CacheEvict`) 로직이 없습니다.**
- **개선안**: 때문에 이번 맵 사태처럼 "DB를 바꿨는데 화면엔 적용이 안 되는" 현상이 생깁니다. 관리자 페이지 등에서 데이터를 건드리면 캐시가 자동으로 비워지도록 어노테이션들을 보강해야 합니다.

**4. MyBatis Mapper.xml의 SQL 중복 심화**
- **현상**: `CollectionMapper.xml`을 보면 `<select>` 문마다 `SELECT id, name, location...` 처럼 컬럼명을 일일이 수십 번 반복해서 적어두었습니다. 이번에 `event_name` 하나를 추가할 때 엄청나게 많은 쿼리를 일일이 고쳐야 했습니다.
- **개선안**: `<sql id="Base_Column_List">...` 덩어리를 만들어 조각조각 재사용하게 만들면, 앞으로 속성 하나를 추가할 때 파일 상단에서 단 1줄만 고치면 전부 적용되게 바꿀 수 있습니다.

---

### 🚨 프론트엔드(HTML/JS) 측면의 문제점

**1. Javascript 파일 비대화 및 모듈화 부재**
- **현상**: 지도 로직을 담당하는 `map.js`가 무려 1,000줄이 넘습니다! 마커를 찍는 로직, 팝업 띄우는 로직, 카테고리 렌더링 로직이 한 파일에 섞여 있어 추후 에러를 잡기 힘들어집니다.
- **개선안**: ES6 모듈을 활용하여 `map-data.js`, `map-ui.js`, `map-markers.js` 3개 정도로 파일을 찢어서 관리해야 합니다.

**2. 하드코딩된 API 엔드포인트 및 이미지 경로**
- **현상**: JS 파일이나 자바 파일 곳곳에 `/images/collections/...` 같은 경로 문자열이나 데이터가 박혀(하드코딩 되어) 있습니다. 
- **개선안**: 전역 환경 설정이나 프로퍼티 파일(`application.yml`)에서 경로들을 끌어와서 쓰도록 바꾸면, 향후 서버 주소나 이미지 저장 폴더(CDN/S3 등)가 통째로 바뀌더라도 한 곳만 고치면 해결됩니다.

## 4. 깊은 곳에 숨어있던 치명적인 결함 (추가 발견)

설정 파일(`SecurityConfig.java`, `application.properties` 등)을 아주 깊게 파본 결과, 겉으로는 티가 나지 않지만 **보안과 성능 면에서 바로 서버를 다운시키거나 해킹당할 수 있는 치명적 문제 4가지**를 추가로 발견했습니다!

### 🚨 [보안] CSRF 토큰 비활성화로 인한 해킹 취약점
- **현재**: `SecurityConfig.java` 23번째 줄에서 `/wiki/map/api/**` (지도 조작 API)에 대해 CSRF 방어를 완전히 꺼두었습니다.
- **위험성**: 관리자가 로그인해 둔 상태(세션 유지)로 인터넷 서핑을 하다가 해커가 심어둔 링크를 누르면, 나도 모르는 사이에 내 권한으로 지도의 핀이 싹 다 지워지거나 조작되는 서프(CSRF) 공격에 100% 무방비로 뚫립니다.
- **해결책**: API라도 세션 쿠키 기반 인증을 쓴다면 끄면 안 됩니다. 프론트엔드(자바스크립트)에서 POST/DELETE 요청을 보낼 때 CSRF 토큰을 헤더에 실어 보내도록 코드를 손봐야 합니다.

### 🚨 [성능] 운영 환경인데 템플릿 캐시가 꺼져 있음
- **현재**: `application.properties` 55번째 줄에 `spring.thymeleaf.cache=false` 로 설정되어 있습니다. (주석은 운영 환경 세팅이라고 적혀있는데 값이 반대로 들어가 있습니다)
- **위험성**: 이러면 접속자가 들어올 때마다 매번 하드디스크에서 HTML 파일을 통째로 새로 읽어와서 조립합니다. 동접자가 10명만 넘어가도 디스크 I/O가 폭발해서 핑이 박살 나고 서버가 터집니다.
- **해결책**: 운영 프로파일(`.env`나 서버 세팅)에서는 무조건 `true`로 켜서 HTML 틀을 메모리에 올려두게 튜닝해야 합니다.

### 🚨 [안정성] 공통 에러 처리 방파제(Global Exception Handler) 부재
- **현재**: 사용자가 존재하지 않는 물고기 이름을 검색하거나, DB 조회 중 에러가 났을 때 잡아줄 방어막(`@ControllerAdvice`)이 없습니다.
- **위험성**: 에러가 나면 흉측한 스프링 부트의 기본 '하얀 에러 페이지(Whitelabel Error Page)'와 내부 스택 트레이스 코드가 방문자에게 여과 없이 노출되어 신뢰도를 박살냅니다.
- **해결책**: 에러가 나면 예쁜 팝업이나 "페이지를 찾을 수 없습니다" 화면으로 돌려주는 공통 에러 핸들러 클래스를 하나 만들어야 합니다.

### 🚨 [설계] 관리자 계정을 코드에 때려 박음 (InMemory Authentication)
- **현재**: 시큐리티 설정에서 관리자 아이디와 비밀번호를 텍스트로 딱 1개 설정해서 메모리에 올려 쓰고 있습니다 (`InMemoryUserDetailsManager`).
- **불편함**: 나중에 부매니저를 한 명 더 두거나, 비밀번호를 바꾸고 싶을 때마다 코드를 고쳐서 '서버를 다시 전체 배포'해야 합니다.
- **해결책**: `admin_users` 같은 DB 테이블을 파서 거기다 계정들을 넣고 시큐리티가 DB를 읽어오게 하는 것이 정석입니다.

## 5. 밑바닥까지 탈탈 털어본 최종 아키텍처(구조) 결함 리스트

"더 없느냐"는 말씀에 자극을 받아, 컨트롤러 내부의 파라미터 검증 과정과 테스트 폴더까지 프로젝트의 영혼까지 끌어모아 싹 다 뒤져보았습니다. **실무 기준(Production-ready)으로 볼 때 가장 크게 발목을 잡을 "기본기" 부족 현상** 3가지를 마저 찾아냈습니다.

### 🚨 [안정성] API 데이터 검증(Validation) 로직 완전 부재
- **현재**: `MapController` 등에서 클라이언트가 보내는 데이터를 받을 때(`@RequestBody MapPin pin`) 값이 비어있는지, 좌표가 이상한 문자인지 검증하는 `@Valid` 어노테이션이나 방어 코드가 0줄입니다.
- **위험성**: 누군가 해킹 툴을 써서 `{"mapX": "천", "mapY": 99999999}` 같은 쓰레기 데이터를 POST로 쏘면, 서버가 500 에러를 뿜으며 멈칫하거나 DB에 쓰레기 데이터가 그대로 적재되어 지도가 영원히 깨집니다.
- **해결책**: DTO 수준에서 `@NotNull`, `@Min`, `@Max` 등의 검증을 빡세게 걸고 튕겨내야 합니다.

### 🚨 [표준화] 제멋대로인 API 응답 형식 (JSON 응답 구조 파편화)
- **현재**: 어떤 API는 `List<MapPin>` 자체를 던지고, 어떤 API는 `Map.of("success", true, "id", id)` 처럼 그때그때 기분 내키는 대로 날것의 형식을 조립해서 던집니다.
- **불편함**: 나중에 프론트엔드나 모바일 앱 개발자가 이 API를 가져다 쓸 때, 성공/실패 여부를 판단하는 기준 구문이 API마다 다 달라서 큰 고통을 겪게 됩니다.
- **해결책**: `{ "status": 200, "message": "성공", "data": { ... } }` 형태를 강제하는 만능 `ApiResponse<T>` 객체를 만들어 전체 백엔드 응답을 하나로 통일해야 합니다.

### 🚨 [품질] 자동화 테스트 코드(Test Coverage) 0%
- **현재**: `src/test/java/` 폴더를 열어본 결과, 스프링 부트가 처음 만들어준 껍데기 파일 1개만 덩그러니 방치되어 있고 서비스나 컨트롤러, DB 결함 테스트 코드가 단 한 줄도 없습니다.
- **위험성**: 이번처럼 코드를 예쁘게 고친다고(리팩토링) 손을 댔다가 다른 멀쩡하던 지도나 채집물 기능이 박살 났을 때, 개발자가 화면에 직접 들어가서 보기 전까진 아무도 에러가 난 줄 모릅니다. 언제 터질지 모르는 시한폭탄과 같습니다.
- **해결책**: 최소한 핵심 서비스 로직(`CollectionService`, `MapPinService`)이라도 Junit5를 이용해 `Mockito` 테스트 코드를 짜두면 빌드할 때 에러를 미리 잡아줍니다.

### 🚨 [운영 관리] 서버 에러 및 추적용 로깅(Logging) 로직 전무
- **현재**: 모든 백엔드 `Controller`와 `Service` 자바 클래스에 `@Slf4j` 어노테이션이나 `log.info()`, `log.error()` 같은 로그 기록 코드가 단 한 줄도 없습니다.
- **위험성**: 사용자가 지도를 조작하다가 서버에서 500 에러 배를 뿜었을 때, 나중에 서버 콘솔 로그(`app.log`)를 까봐도 "언제 누가 무슨 파라미터를 던져서 터졌는지" 단서가 하나도 남지 않아 에러 해결 시 100% 심증과 감에만 의존해야 합니다.
- **해결책**: 핵심 요청과 예외 발생 시점에는 무조건 로그를 찍도록 설정해야 합니다.

### 🚨 [데이터 정합성] 핀 조작 서비스 트랜잭션(`@Transactional`) 누락
- **현재**: 지도의 핀을 삽입(INSERT), 수정(UPDATE), 삭제(DELETE)하는 막중한 권한을 가진 `MapPinService.java`에 `@Transactional` 선언이 쏙 빠져 있습니다. (반면 `WikiReportService` 같은 파일에는 존재함)
- **위험성**: 지금 당장은 핀을 하나씩 지워서 마이바티스(MyBatis) 자동 커밋으로 버티고 있지만, 만약 나중에 "선택한 핀 10개 한 번에 지우기" 같은 복합 로직이 들어갔을 때 5번째에서 에러가 터지면, 앞의 4개만 지워지고 나머지는 영원히 DB에 떠도는 고아(데이터 불일치) 상태가 됩니다.
- **해결책**: 데이터를 변경(`INSERT/UPDATE/DELETE`)하는 모든 Service 메서드나 클래스 상단에는 무조건 `@Transactional`을 붙여서 일관성을 유지해야 합니다.

## 6. 전체 스캔 후 총평 (결론)

요청하신 대로 **현재 프로젝트의 단 하나의 파일도 빠짐없이** (설정, 컨트롤러, 서비스, 템플릿, 자바스크립트 등) 전부 까발려 보았습니다. 

**긍정적인 부분 (잘된 점):**
- Thymeleaf 템플릿에서 `th:utext` 같은 위험한 문법을 쓰지 않아 프론트엔드 XSS(크로스 사이트 스크립팅) 공격에는 안전합니다.
- MyBatis SQL 쿼리들 전부 파라미터 바인딩(`# {}`)이 완벽하게 되어 있어, 치명적인 SQL 인젝션 해킹에는 무적입니다.
- `ReportController`(제보하기) 쪽을 보면 꽤 훌륭하게 데이터 검증(`@Valid`)과 HTML 이스케이프가 잘 구현되어 있습니다.

**가장 큰 문제점 (사일로 현상 및 파편화):**
- **내부 코딩 컨벤션(표준화)의 부재**가 가장 큽니다. 제보 컨트롤러에서는 검증을 빡세게 하지만 맵 컨트롤러에서는 아무것도 안 하고, 어떤 코드는 레코드를 쓰고 어떤 코드는 엔티티를 통으로 던집니다. 
- 개발(AI가 짜준 기능 위주)에만 몰두하다 보니, 전체 시스템을 하나로 아우르는 뼈대(Global Error Handler, API Response 표준화, 공통 캐시 전략, 로깅, 트랜잭션, 테스트)가 모래성처럼 약합니다.

**👉 핵심 요약:**
기능은 아주 잘 동작합니다. 하지만 이 상태로 라이브(운영) 서버에 올린 뒤 유저 트래픽이 몰리기 시작하면, 며칠 못 가 서버가 뻗거나, 해킹당하거나, 원인을 알 수 없는 에러 로그 무덤에서 밤을 새우게 될 확률이 매우 높습니다. 가장 급한 보안(`CSRF`)과 성능(`템플릿 캐시`) 결함부터 즉시 조치하시기를 바랍니다.
